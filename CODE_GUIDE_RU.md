# MicroLearn: гайд по коду простыми словами

Этот файл объясняет проект MicroLearn очень простым языком. Он нужен, чтобы быстро понять, где что лежит, как части проекта разговаривают друг с другом, и что можно сказать на защите, если комиссия спросит про код.

## Самая короткая идея проекта

MicroLearn — это учебная LMS-платформа. LMS можно представить как онлайн-школу:

- студент заходит на сайт;
- выбирает курс;
- записывается;
- проходит уроки;
- сдаёт задания и тесты;
- получает сертификат;
- преподаватель смотрит прогресс и проверяет работы;
- админ следит за пользователями, жалобами, сертификатами и событиями.

Проект не пытается быть настоящим большим SaaS вроде Coursera. Это дипломный MVP: главное — показать рабочую логику обучения.

## Как всё устроено на бытовом примере

Представим, что MicroLearn — это учебный центр.

Frontend — это витрина и залы учебного центра. Пользователь видит страницы, кнопки, формы, курсы, кабинеты.

Backend — это администрация за стойкой. Она принимает заявки, проверяет права, говорит базе данных, что сохранить, и отдаёт ответ обратно на сайт.

Database — это склад и архив. Там лежат пользователи, курсы, уроки, прогресс, задания, тесты, сертификаты.

API route — это окошко, куда frontend обращается с просьбой. Например: “запиши студента на курс”.

Controller/route — это официант. Он принимает заказ от frontend, проверяет, что заказ понятный, передаёт его на кухню и приносит ответ.

Service — это повар или специалист на кухне. Он знает конкретную бизнес-логику: как считать прогресс, как проверить тест, как выдать сертификат.

Prisma — это кладовщик-переводчик. Код говорит Prisma: “найди курс”, а Prisma переводит это в SQL-запрос к PostgreSQL.

Middleware — это охранник на входе. Он проверяет токен, роль пользователя и решает, можно ли идти дальше.

Seed data — это заранее расставленные учебные экспонаты. Пользователи, курсы и задания уже есть, чтобы на защите не заполнять всё вручную.

## Главные папки проекта

### `app`

Это страницы сайта на Next.js App Router.

Пример:

- `app/page.tsx` — главная страница;
- `app/student/page.tsx` — кабинет студента;
- `app/teacher/page.tsx` — кабинет преподавателя;

Если объяснять совсем просто: папка `app` отвечает за адреса сайта. Как таблички на дверях в здании.

### `components`

Здесь лежат куски интерфейса, из которых собираются страницы.

Например:

- `components/ml/nav.tsx` — верхняя навигация;
- `components/ml/search-bar.tsx` — поиск;
- `components/ml/course-detail.tsx` — большая страница курса;
- `components/ml/lesson-reader.tsx` — чтение урока, задания и тесты;
- `components/ml/admin/panel.tsx` — админ-панель;
- `components/ml/teacher/student-analytics.tsx` — аналитика преподавателя по студентам.

Компонент — это как деталь конструктора. Страница берёт несколько деталей и собирает из них экран.

### `lib`

Это общие помощники для frontend.

Главные файлы:

- `lib/api.ts` — клиент для запросов к backend;
- `lib/auth-context.tsx` — хранит состояние входа пользователя;

Если `components` — это кнопки и экраны, то `lib` — это провода за стеной.

### `backend`

Это серверная часть. Она отвечает за данные, безопасность, роли и бизнес-логику.

Внутри backend есть несколько важных папок.

Здесь описана база данных.

Главный файл:

Файл `backend/prisma/seed.ts` создаёт demo-данные. Это как заранее подготовленный стенд для защиты.

Примеры:

- `lessons.ts` — материалы урока;
- `enrollments.ts` — запись студента на курс;
- `progress.ts` — завершение урока и прогресс;
- `assignments.ts` — задания и сдача домашних работ;

Route — это официант: принял запрос, проверил, передал дальше, вернул результат.

### `backend/src/services`

Здесь лежит логика, которую не хочется держать прямо в routes.

Примеры:

- `progress.service.ts` — считает прогресс курса;
- `certificate.service.ts` — создаёт сертификат;
- `quiz.service.ts` — проверяет ответы теста;
- `adaptive.service.ts` — находит слабые темы по quiz и планирует повторение карточек;
- `plan.service.ts` — решает, какой тариф что разрешает;
- `audit-log.service.ts` — записывает важные действия в журнал;
- `token.service.ts` — создаёт и проверяет JWT-токены.

Service — это специалист. Например, route спрашивает: “Можно ли студенту записаться?”, а service отвечает по правилам тарифа.

### `backend/src/middleware`

Middleware — это проверяющие между запросом и логикой.

Примеры:

- `auth.ts` — проверяет access token и роль пользователя;

Аналогия: охранник и контролёр билетов.

## Как пользовательский сценарий проходит через проект

Пример: студент записывается на курс.

1. Студент нажимает кнопку на frontend.
2. Компонент вызывает `api.post("/enrollments")`.
3. Запрос приходит в `backend/src/routes/enrollments.ts`.
4. Middleware проверяет, что пользователь вошёл.
5. `plan.service.ts` проверяет, разрешает ли тариф записаться.
6. Prisma сохраняет запись в PostgreSQL.
7. Backend пишет событие в audit log.
8. Frontend получает ответ и обновляет экран.

Если совсем коротко: кнопка -> API -> проверка -> база -> ответ.

## Как работает оплата в MicroLearn

Сейчас в проекте подключён Stripe test mode. Это режим Stripe, в котором можно создать настоящий `PaymentIntent` и показать форму карты через Stripe Elements, но реальные деньги не списываются.

Текущая оплата работает так:

1. На странице тарифов пользователь выбирает план.
2. Frontend отправляет запрос на backend:

```text
POST /api/plans/subscribe
```

3. Backend создаёт Stripe `PaymentIntent` в test mode и возвращает `clientSecret`.
4. Frontend показывает поле карты через Stripe Elements.
5. Пользователь вводит тестовую карту `4242 4242 4242 4242`.
6. Stripe подтверждает test payment.
7. Frontend отправляет `paymentIntentId` на:

```text
POST /api/plans/confirm
```

8. Backend проверяет PaymentIntent в Stripe, обновляет поле `plan` у пользователя и сохраняет `stripe_payment_id` в таблицу `PlanPayment`.

Например:

- `FREE` ограничивает доступ;
- `PRO` даёт больше возможностей;
- `PREMIUM` включает сертификаты.

В коде это находится здесь:

- `backend/src/routes/plans.ts`;
- `backend/src/services/payment.service.ts`;
- `backend/src/services/plan.service.ts`.

На защите это можно объяснить так:

> В дипломном MVP оплата подключена через Stripe test mode. Это безопасный режим: мы создаём настоящий PaymentIntent и показываем Stripe Elements, но используем тестовую карту, поэтому реальные деньги не списываются. После успешного test payment backend обновляет тариф пользователя и сохраняет `stripe_payment_id`.

Фраза “достаточно подменить одну реализацию сервиса” теперь ближе к реальности: работа со Stripe вынесена в `payment.service.ts`, а тарифная логика остаётся в `plan.service.ts`. Для production всё равно понадобятся webhooks, обработка возвратов, подписки и реальные ключи.

> Сейчас это test mode-интеграция Stripe для дипломного MVP. Следующий шаг — добавить Stripe webhooks и production-настройки, но бизнес-логику курсов трогать не нужно.

Такой ответ звучит честно: оплата уже не является простой заглушкой, но всё ещё безопасна для демонстрации.

## Как работает вход

Вход сделан через email/password.

При логине backend выдаёт:

- access token — короткий ключ для запросов;
- refresh token — более долгий ключ в httpOnly cookie.

Access token сейчас хранится на frontend в `localStorage`. Для диплома это допустимо, но в production лучше хранить токены осторожнее, потому что при XSS localStorage можно прочитать.

Что можно сказать:

> Для MVP выбран простой JWT-подход. Refresh token хранится безопаснее через httpOnly cookie, а access token используется клиентом для API-запросов. В production я бы усилил защиту от XSS и рассмотрел session-only/cookie-based подход.

## Как работают задания

Задание привязано к уроку.

Студент:

- открывает урок;
- видит задание;
- пишет ответ;
- отправляет работу.

Преподаватель:

- открывает урок;
- видит сдачи студентов;
- ставит статус;
- пишет feedback;
- может поставить score.

Статусы:

- `SUBMITTED` — работа отправлена;
- `REVIEWED` — проверена;
- `NEEDS_REVISION` — нужна доработка.

Главные файлы:

- `backend/src/routes/assignments.ts`;
- `components/ml/lesson-reader.tsx`;
- `backend/prisma/schema.prisma`.

## Как работают тесты

Quiz привязан к уроку.

Поддерживаются два типа вопросов:

- `SINGLE_CHOICE` — один правильный ответ;
- `MULTIPLE_CHOICE` — несколько правильных ответов.

Студент выбирает ответы, backend проверяет их и создаёт `QuizAttempt`.

Проверка сделана на backend, а не только на frontend. Это важно, потому что frontend можно обмануть, а backend сложнее.

Главные файлы:

- `backend/src/routes/quizzes.ts`;
- `backend/src/services/quiz.service.ts`;
- `backend/src/services/quiz.service.test.ts`;
- `components/ml/lesson-reader.tsx`.

## Как работает Adaptive Microlearning Engine

Adaptive Engine можно представить как внимательного репетитора после теста.

Quiz показывает, где студент ошибся, а Adaptive Engine говорит: “Значит, тему `React Hooks` нужно повторить короткими карточками”. Он не пишет новые тексты и не обращается к AI. Все карточки уже лежат в базе: их заранее создаёт преподаватель или seed-данные.

Что происходит по шагам:

1. У quiz-вопроса есть поле `topic`.
2. Студент отправляет ответы.
3. `quiz.service.ts` проверяет, какие вопросы правильные, а какие нет.
4. `adaptive.service.ts` смотрит темы неправильных вопросов.
5. Prisma обновляет `StudentWeakTopic`: сколько было ошибок, сколько попыток, какой strength score.
6. Backend находит `MicrolearningCard` по этой теме и создаёт `AdaptiveReview`.
7. В кабинете студента появляется Daily Challenge.

Расписание повторения специально простое, чтобы его легко объяснить на защите:

- ошибся -> повторить завтра;
- правильно -> через 3 дня;
- ещё раз правильно -> через 7 дней;
- несколько раз подряд правильно -> через 14 дней.

Для студента это выглядит как короткая тренировка: вопрос карточки, кнопка “Показать ответ”, затем “Знал” или “Ошибся”.

Для преподавателя это аналитика: какие темы чаще всего слабые у студентов по его курсу и какие карточки уже подготовлены.

Стабильный demo-сценарий для защиты:

1. Войти студентом `temir@microlearn.io` / `Password123!`.
2. Открыть `/student/adaptive`.
3. После fresh seed показать стартовые данные: 5 карточек Daily Challenge и 2 слабые темы.
4. Показать темы стартовых карточек: `CSS Layout` — 3 карточки, `TypeScript Types` — 2 карточки.
5. Открыть курс `Intro to Figma`, урок `1.4 Контрольный чеклист`.
6. Пройти quiz с ошибками.
7. Вернуться на `/student/adaptive` и показать, что Daily Challenge и слабые темы обновились по темам ошибок.
8. Войти преподавателем `aigerim@microlearn.io` / `Password123!`.
9. В кабинете преподавателя открыть Adaptive Insights по курсу `Intro to Figma`.

Для этой демонстрации лучше использовать именно `aigerim@microlearn.io`: weak topics из seed привязаны к её курсу `Intro to Figma`. Аккаунт `daniyar@microlearn.io` тоже имеет карточки, но после fresh seed не показывает готовые слабые темы студентов.

Главные файлы:

- `backend/prisma/schema.prisma` — модели `MicrolearningCard`, `StudentWeakTopic`, `AdaptiveReview`;
- `backend/src/services/adaptive.service.ts` — расчёт слабых тем и расписания;
- `backend/src/routes/adaptive.ts` — API для студента и преподавателя;
- `components/ml/student/adaptive-daily-challenge.tsx` — блок в кабинете студента;
- `components/ml/student/adaptive-practice.tsx` — экран прохождения карточек;
- `components/ml/teacher/adaptive-insights.tsx` — преподавательская аналитика и карточки.

## Как работают сертификаты

Сертификат создаётся после завершения курса для Premium-пользователя.

У сертификата есть:

- `verificationCode`;
- статус `VALID` или `REVOKED`;
- публичная страница проверки.

Публичная проверка:

```text
/certificates/verify/ML-DEMO-2026
```

Админ может отозвать сертификат. Тогда страница проверки покажет `REVOKED`.

Главные файлы:

- `backend/src/routes/certificates.ts`;
- `backend/src/services/certificate.service.ts`;
- `app/certificates/verify/[code]/page.tsx`;
- `components/ml/certificates/verify.tsx`.

## Как работает audit log

Audit log — это журнал важных действий.

Он записывает события вроде:

- студент записался на курс;
- студент завершил урок;
- студент отправил задание;
- преподаватель проверил работу;
- студент прошёл тест;
- adaptive engine нашёл слабую тему;
- студент прошёл карточку повторения;
- пользователь сменил тариф;
- админ отозвал сертификат.

Аналогия: журнал у администратора школы, куда записывают важные события.

Главные файлы:

- `backend/src/services/audit-log.service.ts`;
- `backend/src/routes/admin.ts`;
- `components/ml/admin/panel.tsx`.

## Что можно сказать комиссии про архитектуру

Короткий вариант:

> Проект разделён на frontend и backend. Frontend отвечает за интерфейс, backend — за бизнес-логику и доступ к базе. Prisma используется как слой работы с PostgreSQL. Роли пользователей проверяются на backend через middleware. Основные действия логируются в audit log.

Человеческий вариант:

> Я старался не складывать весь код в один файл. Страницы лежат отдельно, компоненты отдельно, API-роуты отдельно, а сложная логика вынесена в services. Так проект проще объяснять, поддерживать и расширять.

## Честные косяки и планы на будущее

Это не минусы, которые ломают проект. Это нормальные ограничения MVP.

1. Production-оплата пока не подключена.

Тут я сделал Stripe test mode: форма карты и PaymentIntent настоящие, но используются тестовые ключи и карта `4242 4242 4242 4242`. Для production нужно будет включить реальные ключи, webhooks и юридическую часть оплаты.

2. Google/OAuth вход пока не сделан.

Сейчас вход работает через email/password. OAuth можно добавить позже, но для диплома главная цель — LMS-функциональность: курсы, уроки, задания, тесты, прогресс, сертификаты.

3. `prisma db push` используется для demo-запуска.

Для production лучше использовать полноценные миграции Prisma. В MVP `db push` удобнее, потому что быстро поднимает demo-базу.

4. Access token хранится в `localStorage`.

Это проще для учебного проекта. В production я бы усилил безопасность и сделал более строгую схему хранения токенов.

5. Audit log учебный, не enterprise SIEM.

Он показывает идею журналирования действий, но это не полноценная система безопасности уровня банка.

6. Payment layer пока простой.

Работа со Stripe вынесена в `payment.service.ts`, но полноценного `PaymentProvider`-интерфейса для разных провайдеров пока нет. Хороший следующий шаг — добавить adapter для Stripe/CloudPayments и webhooks.

## Самое главное для защиты

Если объяснять проект за одну минуту:

> MicroLearn — это LMS-платформа для коротких курсов. Frontend сделан на Next.js, backend на Express, база PostgreSQL через Prisma. Есть роли студента, преподавателя и администратора. Студент записывается на курс, проходит уроки, сдаёт задания и тесты, получает сертификат. Adaptive Engine анализирует ошибки в quiz и выдаёт короткие повторения по слабым темам без AI. Преподаватель проверяет работы и видит аналитику. Админ управляет пользователями, жалобами, сертификатами и audit log. Оплата работает через Stripe test mode: можно показать PaymentIntent и карту 4242, но реальные деньги не списываются.
